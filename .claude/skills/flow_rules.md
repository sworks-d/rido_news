# flow_rules.md

## RIDO組織全体の情報循環ルール

全エージェントがこのファイルを参照して
指示・報告・改善の流れを把握する。

---

## 1. 基本原則

### 意思決定は上から下へ
方向性・指示・ミッションは必ず上位レイヤーから渡される。
下位レイヤーが勝手に方向性を変えない。

### 報告・改善提案は下から上へ
実行結果・精度データ・問題検知は必ず上位レイヤーに上げる。
飛び越えて報告しない。

### 改善ループは必ず承認を通す
skillsファイルの変更はリーダーの承認なしに実行しない。

---

## 2. 指示の流れ（上から下）

```
俺（オーナー）
  ↓ 方向性・承認（Discord）
参謀
  ↓ 戦略・方針
艦長
  ↓ コンテンツ方針・ミッション
部隊長
  ↓ 具体的な指示・タスク
パイロット / 整備士
  ↓ 装備・参照
モビルスーツ（skills）
```

---

## 3. 報告の流れ（下から上）

```
パイロット
  ↑ 実行ログ・実行結果を部隊長に報告
部隊長
  ↑ 精度サマリー・問題検知を艦長に報告
艦長
  ↑ 全部隊サマリー・改善提案を参謀に報告
参謀
  ↑ 週次レポート・停滞・異常をリーダーにDiscord送信
リーダー
  → 承認・次の方向性を判断
```

---

## 4. 改善ループ

```
部隊長が精度問題を検知
  ↓
部隊長が整備士に改善依頼
  ↓
整備士がskills更新案を作成
  ↓
艦長が内容を確認・承認
  ↓
リーダーが最終承認（Discord）
  ↓
整備士がskillsを更新・知見をインデックス化
  ↓
パイロットの精度が上がる
  ↓（ループ）
```

---

## 5. 報告フォーマット

### パイロット → 部隊長（実行後即時）
```json
{
  "agent": "news_writer",
  "executed_at": "2026-04-02T06:00:00Z",
  "total": 12,
  "passed": 10,
  "rejected": 2,
  "rejection_reasons": ["命令調", "感嘆符超過"],
  "anomaly": null
}
```

### 部隊長 → 艦長（日次・毎朝7時）
```json
{
  "commander": "creative",
  "date": "2026-04-02",
  "pilots": {
    "news_writer": { "precision": 0.83, "trend": "+2%" },
    "route_writer": { "precision": 0.78, "trend": "-3%" },
    "spot_writer":  { "precision": 0.91, "trend": "+1%" }
  },
  "issues": ["route_writerの精度低下・引用改変が再発"],
  "mechanic_request": true
}
```

### 艦長 → 参謀（日次・毎朝8時）
```json
{
  "date": "2026-04-02",
  "overall_precision": 0.84,
  "published": 12,
  "pending": 2,
  "issues": ["route_writerの精度低下"],
  "mechanic_dispatched": true,
  "skills_updated": false
}
```

### 参謀 → リーダー（週次・月曜8時・Discord）
```
【週次レポート】〇〇週

■ 全体精度：84%（先週比+2%）
■ 配信実績：84件
■ 承認待ち：3件
■ skills更新：2件

■ 今週の改善
route_writerの引用改変を修正（precision 78%→85%）

■ 停滞・要確認
なし

■ 来週の方針提案
東海エリアのスポットデータが増加傾向。
金曜配信の強化を提案します。
→ 承認 / 却下 / 修正
```

---

## 6. 停滞・異常アラート

### 即時アラート条件（参謀がDiscordに送信）
```
エージェントが停止した
Layer 1引っかかりが連続3件以上
同じミスが3回繰り返された
精度が前週比-10%以上低下
```

### アラートフォーマット
```
【緊急アラート】
対象：route_writer
問題：引用改変が3件連続発生
状況：整備士に改善依頼済み
対応：skills/route_write.mdの更新案を作成中
→ 確認をお願いします
```

---

## 7. 飛び越え禁止ルール

```
NG: パイロットが直接参謀に報告する
NG: パイロットが直接整備士に依頼する
NG: 部隊長がリーダーに直接報告する
NG: 整備士がリーダーに直接承認を求める
OK: 緊急時は参謀が飛び越えてリーダーに即時報告
```

---

## 8. 知見のインデックス化

整備士がskillsを更新したら必ずSupabaseに記録する。

```json
{
  "file": "skills/route_write.md",
  "problem": "quoted_commentの改変",
  "knowledge": "改変禁止を制約に追加・具体例を付与",
  "source": "route_writer",
  "effect": "precision 78%→85%",
  "timestamp": "2026-04-07T10:00:00Z"
}
```

---

## 9. 監視エージェント（monitor）

### 役割
全パイロットの状態をstatus-board.mdで常時監視する。
異常を検知したら参謀にアラートを送る。

### status-board.mdの更新ルール
```
実行開始時：状態を「🔄 実行中」に更新
実行完了時：状態を「✅ 完了」・件数を記録
実行失敗時：状態を「❌ 失敗」・備考に理由を記録
翌日の実行開始前に「⏳ 待機」にリセット
```

### アラートの流れ
```
monitor が異常検知
        ↓
参謀のagent_statusに書き込む
        ↓
参謀がDiscordにアラート送信
        ↓
リーダーが判断
```
